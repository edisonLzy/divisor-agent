# 前端技术文档（MVP，Electrobun 方案）

## 1. 文档目标

本文档定义 divisor-agent App 端在 MVP 阶段的目标实现方案。结论如下：

- **桌面宿主框架改为 Electrobun**，不再使用 Tauri。
- **Agent Core 运行在本地 App 内**，而不是运行在 Server。
- **Server 仅负责历史会话与元数据管理**，不负责 Agent Loop。
- **App 内不再需要 ACP WebSocket**；本地进程间通信改为 **Electrobun Typed RPC**。
- **Extensions 机制以本地 Bun 主进程为宿主**，优先兼容 `~/.pi/extensions` 的内容加载方式。

---

## 2. 技术栈

| 层级 | 技术 |
| :--- | :--- |
| 桌面宿主 | Electrobun |
| 本地主进程 | Bun + TypeScript |
| UI 框架 | React 19 + TypeScript |
| WebView 构建 | Electrobun 视图构建 |
| WebView 与主进程通信 | Electrobun Typed RPC |
| 与服务端通信 | tRPC v11 + `@trpc/react-query` + `@tanstack/react-query` v5 |
| 样式 | Tailwind CSS v4 |
| 富文本输入框 | TipTap |
| 流式 Markdown 渲染 | `streamdown` 或同类支持流式修复的渲染器 |
| 本地 Agent 运行时 | `@mariozechner/pi-agent-core` |
| 本地工具执行 | Bun API（文件、进程、网络） |

## 3. 架构总览

### 3.1 总体架构

```text
┌─────────────────────────────────────────────────────────────────────┐
│ App (Electrobun)                                                   │
│                                                                     │
│  ┌───────────────────────┐        Typed RPC        ┌──────────────┐ │
│  │ React WebView         │ ◄────────────────────► │ Bun Main     │ │
│  │                       │                        │ Process      │ │
│  │ - SessionTree         │                        │              │ │
│  │ - ChatView            │                        │ - Agent Loop │ │
│  │ - InputBar            │                        │ - Extensions │ │
│  │ - ApprovalDialog      │                        │ - FS/Terminal│ │
│  └──────────┬────────────┘                        │ - HITL       │ │
│             │                                     └──────┬───────┘ │
│             │ tRPC / HTTP                                  │         │
└─────────────┼──────────────────────────────────────────────┼─────────┘
              │                                              │
              ▼                                              ▼
      ┌──────────────────┐                          ┌──────────────────┐
      │ Server           │                          │ Local OS         │
      │                  │                          │                  │
      │ - sessions CRUD  │                          │ - file system    │
      │ - history store  │                          │ - terminal       │
      │ - models meta    │                          │ - user home dir  │
      └──────────────────┘                          └──────────────────┘
```

### 3.2 关键结论

1. **实时对话链路完全本地化**：React 通过 Electrobun RPC 调用 Bun 主进程，主进程直接运行 Agent。
2. **会话历史仍在 Server**：会话树、历史消息、Fork 元数据、模型列表继续由 Server 提供和持久化。
3. **不再使用 ACP WebSocket**：此前 ACP 的价值在于“远程 Agent + 本地工具”桥接；Agent Loop 迁到 App 后，这一层不再必要。
4. **Extensions 在本地加载**：扩展运行环境必须靠近 Agent Core，才能直接注入工具、命令、上下文能力。

## 4. 职责边界

### 4.1 React WebView 负责

- UI 渲染：会话树、消息流、审批弹窗、设置页。
- 通过 tRPC 读取和变更 Server 中的会话元数据。
- 通过 Electrobun RPC 向 Bun 主进程发送用户 prompt 与审批结果。
- 监听 Bun 主进程推送的流式消息、审批请求、Fork 完成事件。

### 4.2 Bun 主进程负责

- 初始化并持有 Agent 实例。
- 加载内置工具与本地 Extensions。
- 执行本地文件系统、终端等工具调用。
- 维护审批工作流。
- 将用户消息、助手消息、Fork 结果等通过 HTTP/tRPC 回写到 Server。
- 将流式结果实时推送给 WebView。

### 4.3 Server 负责

- 会话树管理：`list/create/rename/delete/fork`
- 历史消息持久化与查询：`history/append`
- 模型列表与应用配置读取
- 可选的用户级同步、审计、后续云端能力

### 4.4 明确不做的事

- WebView 不直接访问本地文件系统或终端。
- Server 不直接运行 Agent Loop。
- App 内不维护 ACP 协议兼容层作为主路径。

## 5. 为什么选择 Electrobun

### 5.1 与当前目标的匹配度

本项目当前的核心诉求不是“Rust 原生能力”，而是：

- 让 **pi-agent-core 在本地直接运行**
- 最大化复用 **Node/Bun 生态**
- 本地加载并执行 **Extensions**
- 以较低改造成本承接现有 React UI

Electrobun 更适合这个方向，因为：

- 主进程就是 **Bun/TypeScript**
- 更容易承接 Node 风格的 Agent/Extension 运行时
- 不需要再通过 Rust 做一次本地能力代理
- Bun 对文件、进程、HTTP、脚本执行的 API 更直接

### 5.2 相比 Tauri 的取舍

| 维度 | Electrobun | Tauri |
| :--- | :--- | :--- |
| 本地 Agent 运行 | 直接在 Bun 中运行 | 需要 Rust 包一层或额外 Node 子进程 |
| Node/Bun 生态兼容 | 高 | 中 |
| Extensions 实现难度 | 低 | 高 |
| 本地工具接入 | 直接用 Bun API | 需 Rust Command / Plugin |
| 包体与原生资源占用 | 通常更高 | 通常更低 |
| 安全默认面 | 需靠应用层收紧 | 宿主能力边界更强 |

当前 MVP 更优先 **Agent 运行效率与扩展能力**，因此选择 Electrobun。

## 7. 运行时设计

### 7.1 启动流程

1. Electrobun 启动 Bun 主进程。
2. Bun 主进程创建 BrowserWindow / BrowserView。
3. Bun 主进程初始化：
   - 配置加载
   - 模型解析
   - Extensions 扫描与注册
   - Agent Runtime 初始化
4. WebView 加载 React 应用。
5. React 初始化：
   - 建立 Electrobun RPC 客户端
   - 建立 tRPC Client
   - 拉取会话树、模型列表、当前设置

### 7.2 Prompt 执行链路

```text
User Input
  → React InputBar
  → Bun RPC: sessionPrompt
  → Bun Agent Runtime
  → Agent 调用 LLM / 执行工具 / 请求审批
  → Bun 推送 chunk 到 WebView
  → Bun 回写消息到 Server
  → React 渲染最终结果
```

### 7.3 Fork 链路

```text
React 触发 Fork
  → Server sessions.fork
  → 返回 newSessionId
  → React 切换到新会话
  → Bun 后续对 newSessionId 继续 prompt
```

说明：

- **Fork 元数据和历史复制由 Server 完成**
- Bun 主进程只消费新的 `sessionId`

### 7.4 审批链路

```text
Agent 请求高风险工具
  → Bun permission-service 拦截
  → Bun 推送 sessionRequestPermission 到 WebView
  → React 弹出 ApprovalDialog
  → 用户 approve/reject
  → React 调用 Bun RPC 返回决策
  → Bun 恢复工具执行或拒绝
```

## 8. IPC / RPC 设计

### 8.1 设计原则

- WebView 和 Bun 之间只使用 **Typed RPC**
- 所有事件名、参数、返回值都集中定义在 `src/shared/ipc-types.ts`
- WebView 不感知 Bun 内部 Agent/Extension 实现细节

### 8.2 WebView → Bun

| RPC | 参数 | 说明 |
| :--- | :--- | :--- |
| `sessionPrompt` | `{ sessionId, content, model? }` | 发送用户消息 |
| `permissionApprove` | `{ requestId }` | 批准高风险操作 |
| `permissionReject` | `{ requestId }` | 拒绝高风险操作 |

### 8.3 Bun → WebView

| Message | 载荷 | 说明 |
| :--- | :--- | :--- |
| `agentMessageChunk` | `{ type, delta, chunkIndex, sessionId }` | 流式消息块 |
| `agentMessageDone` | `{ sessionId }` | 当前回复完成 |
| `sessionRequestPermission` | `{ requestId, operation, params }` | 触发审批 |
| `sessionForked` | `{ newSessionId, parentSessionId }` | Fork 完成通知 |

### 8.4 不放进 RPC 的内容

以下内容继续由 React 直连 Server：

- `sessions.list`
- `sessions.history`
- `sessions.rename`
- `sessions.delete`
- `sessions.fork`
- `models.list`

## 9. Server 通信设计

### 9.1 React → Server

通过 tRPC 获取：

- 会话树
- 会话历史
- 模型列表
- 重命名 / 删除 / Fork

### 9.2 Bun → Server

Bun 主进程需要直接调用 Server 接口，用于：

- 确保 session 存在
- 追加 user message
- 追加 assistant message
- 写入工具结果
- 上报 Fork 后的会话切换信息（如需要）

### 9.3 推荐接口补充

当前 Server 若仅暴露给 React 查询，还需补充适合 Bun 写入的接口，例如：

| 接口 | 用途 |
| :--- | :--- |
| `sessions.create` | 创建新会话 |
| `sessions.appendMessage` | 追加消息 |
| `sessions.fork` | 基于父会话创建分支 |
| `sessions.touch` | 更新最后活跃时间 |

## 10. Agent Runtime 设计

### 10.1 运行位置

Agent Runtime 固定运行在 Bun 主进程中。

原因：

- 能直接访问 Extensions
- 能直接访问本地工具
- 不需要 ACP 转发
- 用户审批链路更短

### 10.2 运行时职责

- 创建并缓存 session 对应的 Agent 实例
- 注入工具与扩展能力
- 订阅 `message_update` 等事件
- 将 `text_delta` / `thinking_delta` 推送到 WebView
- 将完整消息持久化到 Server

### 10.3 Session 状态管理

主进程需维护：

```ts
interface SessionRuntimeState {
  agent: Agent;
  pendingPermissions: Map<string, PendingPermission>;
  activeModel?: {
    providerId: string;
    modelId: string;
  };
}
```

约束：

- 每个 `sessionId` 对应一个运行时状态
- 可做惰性创建
- 长时间不活跃会话可回收

## 11. 本地工具系统设计

### 11.1 内置工具

MVP 至少提供：

- `fs/read_text_file`
- `fs/write_text_file`
- `terminal/create`

### 11.2 实现方式

使用 Bun API 实现：

- 文件读取：`Bun.file(path).text()`
- 文件写入：`Bun.write(path, content)`
- 终端执行：`Bun.spawn(...)`

### 11.3 安全约束

- 路径沙箱：仅允许工作目录或用户授权目录
- 命令黑名单：拦截危险命令模式
- 写文件、终端执行、跨目录访问默认进入审批

## 12. Extensions 机制方案

### 12.1 目标

1. 尽量兼容 `pi-coding-agent` 对 `~/.pi/extensions` 的内容加载方式
2. 允许用户在本机通过新增 extensions 扩展 Agent 能力
3. Extensions 能直接影响本地 Agent Runtime，而不是经由远端转发

### 12.2 为什么必须放在 App 端

如果 Agent Core 在 App，本地 Extensions 也必须在 App：

- 扩展通常要注入工具、提示词、命令或上下文来源
- 这些能力要在 Agent 推理阶段即时可见
- 若放在 Server，会出现“本地 Agent 看不到本地扩展”的断层

### 12.3 扫描路径

建议的扫描优先级：

1. `项目目录/.pi/extensions`
2. `用户目录/.pi/extensions`
3. App 内置扩展目录（只读）

合并规则：

- 同名扩展按优先级覆盖
- 记录来源路径，便于诊断

### 12.4 扩展装载流程

```text
discover extensions
  → read manifest / entry
  → validate schema
  → dynamic import
  → normalize to internal Extension interface
  → register tools / prompts / hooks
  → inject into Agent Runtime
```

### 12.5 内部统一接口

建议在 Bun 侧做一层适配：

```ts
interface AppExtension {
  name: string;
  version: string;
  tools?: AgentTool[];
  systemPrompts?: string[];
  onSessionStart?: (ctx: ExtensionContext) => Promise<void> | void;
}
```

意义：

- 对外兼容 pi 风格扩展
- 对内收敛成 App 自己的运行时接口

### 12.6 兼容策略

兼容分两层：

1. **目录兼容**：支持直接扫描 `.pi/extensions`
2. **运行时兼容**：对常见 extension 导出格式做适配

不建议盲目承诺 100% 二进制级兼容。更现实的做法是：

- 先兼容最常见的 JS/TS 扩展结构
- 对依赖 Node 特殊 API 的扩展做能力声明和兼容性提示

### 12.7 权限模型

扩展能力也要进入权限体系：

- 只读扩展：默认允许
- 写文件 / 执行命令 / 网络访问：按策略审批
- 扩展来源、版本、入口文件需可审计

## 13. UI 设计要求

### 13.1 整体布局

```text
┌──────────────┬─────────────────────────────────┐
│ SessionTree  │ ChatView                        │
│              │                                 │
│              │ MessageList                     │
│              │                                 │
│              │ InputBar                        │
└──────────────┴─────────────────────────────────┘
```

### 13.2 SessionTree

- 数据来源：`trpc.sessions.list.useQuery()`
- 支持树状递归渲染
- 支持选中、高亮、重命名、删除、刷新
- Fork 成功后自动定位到新节点

### 13.3 ChatView

- 初始历史来自 `trpc.sessions.history.useQuery()`
- 实时消息来自 Bun 推送
- `thinking_delta` 默认折叠
- 工具结果支持结构化块渲染
- 渲染层需用组件映射模式，避免硬编码分支

### 13.4 InputBar

- 使用 TipTap
- `Enter` 发送，`Shift + Enter` 换行
- 预留 Slash Commands、文件引用、会话引用扩展位

### 13.5 ApprovalDialog

- 接收 `sessionRequestPermission`
- 显示完整参数和风险说明
- 批准/拒绝均回传 Bun 主进程

```ts
interface AppState {

    requestId: string;
```

## 16. 从 Tauri 迁移到 Electrobun 的实施步骤

### Phase 1：宿主切换

- 移除 Tauri 运行时依赖
- 新建 `packages/app`
- 建立 Electrobun `bun/index.ts` 与 `src/main.tsx`
- 建立共享 `ipc-types.ts`

### Phase 2：通信切换

- 前端 `invoke/listen` 改为 Electrobun RPC
- 删除 Rust ACP Client 相关实现
- 建立 Bun → React 流式推送链路

### Phase 3：Agent 本地化

- 在 Bun 主进程中集成 `pi-agent-core`
- 将 prompt、chunk、审批、工具调用打通
- Bun 对接 Server 写入历史

### Phase 4：Extensions 能力

- 扫描 `.pi/extensions`
- 建立 loader / registry
- 注入 Agent Runtime

### Phase 5：收口与验证

- 处理窗口行为、配置加载、错误提示
- 补齐打包与发布流程
- 验证历史、Fork、审批、扩展、流式渲染

## 17. 风险与缓解

| 风险 | 说明 | 缓解 |
| :--- | :--- | :--- |
| Bun 兼容性 | 某些依赖可能默认按 Node 设计 | 先适配 MVP 主链路，必要时对单点能力做隔离 |
| 扩展不可控 | 用户扩展可能做高风险操作 | 扩展纳入权限与审计体系 |
| 历史一致性 | 流式渲染和最终落库可能短暂不一致 | 以 Server 历史为准，结束后主动刷新 |
| 本地运行安全面扩大 | Agent 与扩展都在本机执行 | 默认拒绝高风险能力，审批前置 |

---

## 18. MVP 验收标准（App 视角）

- [ ] App 可用 Electrobun 正常启动并渲染主界面
- [ ] React 可通过 Electrobun RPC 向 Bun 发送 prompt
- [ ] Bun 中的 Agent 可正常流式推送 `text_delta` / `thinking_delta`
- [ ] 审批弹窗链路完整可用
- [ ] 会话树与历史消息仍能通过 Server 正常读取
- [ ] Fork 后能正确切换到新会话继续对话
- [ ] App 能扫描并加载本机 `.pi/extensions`
- [ ] 至少一个扩展可成功向 Agent 注入工具或提示能力

---

## 19. 最终结论

对于当前项目，**Electrobun + 本地 Bun 主进程运行 Agent Core + Server 仅负责历史与元数据**，是更贴合目标的 App 方案。

它同时满足：

- 本地运行 Agent
- 更容易承接 Node/Bun 生态
- 支持 Extensions 机制
- 保留现有 React UI
- 保留 Server 的历史会话管理职责

这应作为当前 MVP App 端的基准实现方案。
