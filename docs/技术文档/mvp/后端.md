| 层级 | 技术 |

| 生产构建 | tsc -p tsconfig.build.json |

## 2. 职责边界

Node 服务（Server）负责：
- **Agent Loop**：接收用户 Prompt，驱动 LLM 推理与规划，处理工具调用结果，生成回复流。
- **ACP WebSocket 服务端**：维护与 Tauri Rust Client 的长连接，收发 ACP 消息。
- **会话持久化**：写入、读取、维护 `session-map.json` 及消息历史。
- **HTTP API**：提供元数据查询与 CRUD 接口给前端。

Node 服务**不**负责：
- 直接访问本地文件系统或终端（由 Rust 代理执行）。
- UI 渲染或 IPC 管理。

## 3. 架构分层

```
tRPC 路由层              ACP WebSocket 层
    │                          │
    ▼                          ▼
tRPC Router             ACP Message Handler
(Express adapter)             │
    │                          │
    └───────────┬──────────────┘
               ▼
          Session Service
          ├── AgentLoop (pi-agent-core)
          ├── SessionRepository (读写 session-map.json)
          └── ToolDelegateClient (转发工具请求 → Rust)
```

**两套通道职责**：

| 通道 | 协议 | 职责 |
| :--- | :--- | :--- |
| tRPC (query/mutation) | HTTP | 会话元数据读写（列表、历史、设置、重命名、删除） |
| ACP WebSocket | WebSocket | 会话对话流、工具调用委派、Rust 推送 ACP 事件 |

## 4. tRPC API 设计

### 4.1 Procedure 列表

| Procedure | 类型 | Input | 描述 |
| :--- | :--- | :--- | :--- |
| `sessions.list` | query | — | 返回完整会话树结构 |
| `sessions.history` | query | `{ id: string, cursor?: string }` | 返回指定会话历史消息，支持游标分页 |
| `sessions.rename` | mutation | `{ id: string, name: string }` | 重命名会话 |
| `sessions.delete` | mutation | `{ id: string }` | 删除会话节点（可级联删除子节点） |
| `settings.get` | query | — | 返回模型与应用配置 |

> tRPC 挂载路径：`/trpc`，通过 `createExpressMiddleware` 接入 Express v5。

### 4.2 依赖安装

```bash
# server
pnpm add @trpc/server superjson

# app
pnpm add @trpc/client @trpc/react-query @tanstack/react-query
```

### 4.3 Router 骨架（server）

```ts
// src/trpc.ts
import { initTRPC } from '@trpc/server';
import superjson from 'superjson';

export const t = initTRPC.create({ transformer: superjson });
export const router = t.router;
export const publicProcedure = t.procedure;
```

```ts
// src/routers/sessions.ts
import { z } from 'zod';
import { router, publicProcedure } from '../trpc.js';

export const sessionsRouter = router({
  list: publicProcedure.query(async () => { /* 读 session-map.json */ }),
  history: publicProcedure
    .input(z.object({ id: z.string(), cursor: z.string().optional() }))
    .query(async ({ input }) => { /* 读 messages.jsonl，游标分页 */ }),
  rename: publicProcedure
    .input(z.object({ id: z.string(), name: z.string() }))
    .mutation(async ({ input }) => { /* 更新 session-map.json */ }),
  delete: publicProcedure
    .input(z.object({ id: z.string() }))
    .mutation(async ({ input }) => { /* 删除会话目录 */ }),
});
```

```ts
// src/app.ts — 挂载
import { createExpressMiddleware } from '@trpc/server/adapters/express';
import { appRouter } from './routers/index.js';

app.use('/trpc', createExpressMiddleware({ router: appRouter }));
```

**类型导出（给前端共享）：**
```ts
// src/routers/index.ts
export const appRouter = router({ sessions: sessionsRouter, settings: settingsRouter });
export type AppRouter = typeof appRouter;   // 此类型通过包引用或路径别名共享给前端
```

### 4.4 数据结构

#### SessionNode
```ts
interface SessionNode {
  id: string;
  parentId: string | null;
  name: string;
  timestamp: number;
  children?: SessionNode[];
}
```

#### HistoryMessage
```ts
interface HistoryMessage {
  id: string;
  sessionId: string;
  role: 'user' | 'assistant';
  blocks: MessageBlock[];
  timestamp: number;
}

type MessageBlock =
  | { type: 'text'; content: string }
  | { type: 'thinking'; content: string }
  | { type: 'tool_result'; toolName: string; content: string };
```
> 注：针对于用户发送带有 `@某个文件` 的富文本（TipTap 提供），前端需要将其序列化为原生的 JSX/XML 占位标签（如 `<file path="...">`）并以普通的 `text` 块形式提交与持久化，以此保留其在原句中的语境位置不变。

#### GET /api/sessions/:id/history 响应
```ts
interface HistoryResponse {
  messages: HistoryMessage[];
  nextCursor: string | null;
}
```

### 4.5 错误处理

tRPC 使用 `TRPCError` 替代手写 ApiError：
```ts
import { TRPCError } from '@trpc/server';

throw new TRPCError({
  code: 'NOT_FOUND',        // 对应 HTTP 404
  message: 'Session not found',
});
```

常用错误码映射：`NOT_FOUND` / `BAD_REQUEST` / `INTERNAL_SERVER_ERROR` / `UNAUTHORIZED`。前端自动获得类型化的错误对象，无需额外定义 `ApiError` 接口。

## 5. ACP WebSocket 协议

### 5.1 连接管理

- Rust Client 发起 WebSocket 连接，握手时携带 `clientId` 与协议版本号。
- 服务端维护 `clientId → WebSocket` 映射，连接断开后清理资源。
- 协议头字段：`{ version: string, clientId: string }`。

### 5.2 消息格式（基础结构）
```ts
interface AcpMessage {
  type: string;
  sessionId: string;
  messageId?: string;   // 请求/响应配对标识
  payload: object;
}
```

### 5.3 会话类消息

| 消息类型 | 方向 | 说明 |
| :--- | :--- | :--- |
| `session/start` | Client → Server | 建立会话，携带 `sessionId`、工作目录等上下文 |
| `session/prompt` | Client → Server | 用户输入内容，触发 Agent Loop |
| `session/load` | Client → Server | 请求加载历史块（用于逐块回放） |
| `session/fork` | Client → Server | Fork 当前会话，可选携带 `messageId` 截断点 |

**`session/start` payload：**
```ts
{
  sessionId: string;
  parentId?: string;           // Fork 来源（若为 Fork 会话）
  workingDirectory?: string;   // 工作目录
  inheritedSkills?: string[];  // 继承的技能集合
}
```

**`session/prompt` payload：**
```ts
{
  sessionId: string;
  content: string;    // 会话内容
  // 若包含富文本 @ 胶囊，前端需将其内联序列化为 XML 占位符再发送，例如：
  // "帮我对比一下 <file path='/src/App.tsx' /> 和 <file path='/src/utils.ts' />"
}
```

**`session/fork` payload：**
```ts
{
  sessionId: string;    // 父会话 ID
  messageId?: string;   // POI 截断点；缺省则复制完整历史
}
```

**服务端响应 Fork：**
```ts
{
  type: 'session/forked',
  payload: { newSessionId: string }
}
```

### 5.4 流式输出消息

| 消息类型 | 方向 | 说明 |
| :--- | :--- | :--- |
| `agent_message_chunk` | Server → Client | 携带 `text_delta` 或 `thinking_delta` |
| `agent_message_done` | Server → Client | 当前轮次回复完成 |

**`agent_message_chunk` payload：**
```ts
{
  type: 'text_delta' | 'thinking_delta';
  delta: string;
  chunkIndex: number;   // 用于幂等排序
}
```

### 5.5 工具委派消息

| 消息类型 | 方向 | 说明 |
| :--- | :--- | :--- |
| `fs/read_text_file` | Server → Client | 请求 Rust 读取本地文件 |
| `fs/write_text_file` | Server → Client | 请求 Rust 写入本地文件（需审批） |
| `terminal/create` | Server → Client | 请求 Rust 执行终端命令（需审批） |
| `tool/result` | Client → Server | Rust 返回工具执行结果 |
| `tool/error` | Client → Server | Rust 返回工具执行错误 |

**工具请求 payload 示例（`fs/read_text_file`）：**
```ts
{
  toolCallId: string;   // 与 tool/result 配对
  path: string;
}
```

### 5.6 权限类消息

| 消息类型 | 方向 | 说明 |
| :--- | :--- | :--- |
| `session/request_permission` | Server → Client | 通知 Rust 需用户审批（Rust 再转发前端） |
| `permission/approve` | Client → Server | 用户已批准，Rust 执行后回传 |
| `permission/reject` | Client → Server | 用户已拒绝 |

**`session/request_permission` payload：**
```ts
{
  requestId: string;
  operation: 'fs_write' | 'terminal_exec' | 'cross_dir_access';
  params: object;   // 完整操作参数（路径、命令、diff 等）
}
```

## 6. Agent Loop 设计

```
session/prompt 到达
       │
       ▼
  从 content 中使用 AST/正则提取内联的 <file path="..." /> 占位标签
  交给 Rust 读取对应文件的真实源码内容
       │
       ▼
  将闭合标签撑开，替换成包含真实源码的 <file>源码</file> （完美保留句子语境）
       │
       ▼
  加载所有填充好上下文的历史消息
       │
       ▼
  调用 LLM（流式）
       │
  ┌────┴────────────────┐
  │ text_delta          │ tool_call
  ▼                     ▼
推送 agent_message_chunk  构造工具委派消息 → Rust
                              │
                         等待 tool/result
                              │
                         continue → LLM
       │
       ▼
  agent_message_done（本轮结束）
       │
       ▼
  持久化消息到会话存储
```

**架构扩展设计（Extensibility）：**
- **基于 pi-mono Extension 机制的解耦架构**：鉴于 Agent 核心交互基于 `pi-agent-core` (pi-mono) 实现，为避免业务逻辑（如特定文件的解析、权限弹窗等）侵入核心主循环，所有的功能增强**必须严格通过编写 TypeScript Extension 模块（利用官方 `ExtensionAPI`）进行插件化挂载**。规范如下：
  1. **能力扩充**：如需新增让 LLM 调用的本地系统能力，使用 `pi.registerTool()` 注册标准参数 Schema；如需新增供用户单向手动触发的指令（如 `/clear`），使用 `pi.registerCommand()`。
  2. **上下文流式编排**：利用 `pi.on("input")` 或 `pi.on("context")` 钩子进行无损拦截。例如在喂给大模型前，将前端 `session/prompt` 里的 XML 占位符动态替换为真实的文件源码内容，或在此阶段动态注入/压缩会话上下文。
  3. **沙箱权限与安全红线**：高风险行为的卡点统一收口于 **`pi.on("tool_call")`** 拦截器。在该钩子内部向前端推送 `session/request_permission` 等待审批倒计时，若用户拒绝或超时，直接向引擎返回 `{ block: true, reason: '...' }`，严格控制模型的破坏范围。

**关键约束：**
- **内联上下文预装载（Context Hydrator）实现**：针对前端传来的 `<file path="..."/>` 自闭合占位符，Server 层须编写独立的 Extension，在 **`pi.on("input")` 或 `pi.on("context")`** 的生命周期中截获处理。原地撑开占位符、转化为包含源码节点的最终 Prompt 后，再放行进入 LLM 推理计算逻辑。
- **审批工作流边界**：高危工具（涉及擦写磁盘、系统级终端命令等）的权限拦阻，必须独立写在 **`pi.on("tool_call")`** 事件体内。
- 工具调用结果必须与 `sessionId` 和 `toolCallId` 双重绑定，防止多并发模式下出现串会话灾难。

## 7. 会话存储设计

### 7.1 目录结构
```
~/.pi-agent/
├── session-map.json           # 会话树元数据
└── sessions/
    ├── {sessionId}/
    │   ├── meta.json          # 单会话元数据（name, parentId, timestamp）
    │   └── messages.jsonl     # 消息历史（追加写入）
```

### 7.2 session-map.json 格式
```json
{
  "version": "1",
  "sessions": [
    {
      "id": "sess_001",
      "parentId": null,
      "name": "Initial Session",
      "timestamp": 1700000000000
    }
  ]
}
```

### 7.3 Fork 实现逻辑
1. 读取父会话 `messages.jsonl`，按 `messageId` 截断（POI Fork）或完整复制（全局 Fork）。
2. 为新会话分配 `newSessionId`，写入 `session-map.json`（`parentId` 指向父会话）。
3. 将截断后的消息写入新会话 `messages.jsonl`。
4. 新会话的后续消息与父会话**物理隔离**，父会话不受影响。

### 7.4 一致性要求
- `session-map.json` 写入使用原子替换（先写临时文件，再重命名）。
- 消息追加使用 `messages.jsonl`（每行一条 JSON），避免全量读写。
- 版本号字段用于未来协议演进兼容。

## 8. 安全设计

### 8.1 能力边界（由 Rust 执行，Server 策略层声明）
- 工具调用范围仅限 `fs/read_text_file`、`fs/write_text_file`、`terminal/create`。
- 路径校验由 Rust 沙箱执行，Server 仅做业务意图校验。
- 命令黑名单（如 `rm -rf`、危险 SQL）：Server 在构造 `terminal/create` 前过滤，匹配则改为发送 `session/request_permission` 强制审批或直接拒绝。

### 8.2 审批工作流
- Server 发送 `session/request_permission` → Rust 转发 → 前端弹窗 → 用户决策 → `permission/approve` 或 `permission/reject` 回流。
- 收到 `permission/reject` 时，Server 返回 `permission_denied` 错误给 Agent Loop，Agent 须感知并结束或继续（不执行该操作）。

### 8.3 审计日志
- 每次工具委派、权限申请、审批结果均写入结构化日志。
- 日志字段：`sessionId`、`toolCallId`/`requestId`、`operation`、`params`、`decision`、`timestamp`、`result`。
- 日志按会话 ID 可检索。

## 9. 非功能要求

| 指标 | 目标 |
| :--- | :--- |
| 首屏会话树加载 | < 1.5 秒（1k 节点以内）|
| 首 token 可见时间 | < 2 秒（网络正常）|
| ACP 断连自动重连 | Rust 侧负责，Server 须支持幂等重接（同 sessionId 复用） |
| 工具调用失败 | 返回包含 `code` 和 `message` 的结构化错误 |
| 关键链路日志 | 握手、工具调用、审批均有结构化日志 |

## 10. 验收要求（后端视角）

- [ ] `GET /api/sessions` 正确返回树形结构，1k 节点响应 < 1.5 秒。
- [ ] `GET /api/sessions/:id/history` 支持游标分页，消息顺序正确。
- [ ] `session/prompt` 触发 Agent Loop，流式返回 `agent_message_chunk`，本轮结束发送 `agent_message_done`。
- [ ] `fs/read_text_file` 工具调用成功委派并收到 Rust 返回结果，结果与 sessionId 绑定。
- [ ] 写操作与终端执行必须先发 `session/request_permission`，收到 `permission/approve` 后才执行。
- [ ] `permission/reject` 后 Server 正确返回 `permission_denied`，Agent Loop 不中断崩溃。
- [ ] `session/fork`（带/不带 `messageId`）成功创建新会话，父会话消息不受影响。
- [ ] 审批日志可按 sessionId 查询，字段完整。
- [ ] 越权路径访问被记录日志并拒绝（路径沙箱由 Rust 执行，Server 侧验证业务意图）。
