# 后端技术文档（MVP）

## 1. 技术栈

| 层级     | 技术                                 |
| :------- | :----------------------------------- |
| 运行时   | Node.js（ESM，`"type": "module"`）   |
| Web 框架 | Express v5                           |
| API 层   | tRPC v11（端到端类型安全）           |
| 校验     | Zod v4（tRPC input 层复用）          |
| 日志     | pino + pino-http                     |
| 跨域     | cors                                 |
| 环境变量 | dotenv                               |
| 序列化   | Superjson                            |
| 会话存储 | 本地 JSON 文件（`session-map.json`） |
| 开发运行 | tsx --watch                          |
| 生产构建 | tsc -p tsconfig.build.json           |

---

## 2. 职责边界

Node 服务（Server）负责：

- **会话持久化**：写入、读取、维护 `session-map.json` 及消息历史。
- **HTTP API**：提供元数据查询与 CRUD 接口给前端和 App。
- **模型配置管理**：提供模型列表和配置读取。

Node 服务**不**负责：

- **Agent Loop**：Agent 在 App 端本地运行，Server 不参与推理。
- **工具执行**：本地文件系统、终端操作由 App 端处理。
- **实时通信**：使用 HTTP/tRPC 而非 WebSocket。

---

## 3. 架构分层

```
tRPC 路由层
    │
    ▼
tRPC Router
(Express adapter)
    │
    ▼
Session Service
├── SessionService (读写 session-map.json)
└── ModelService (读取 ~/.pi/agent/models.json)
```

**通信通道职责**：

| 通道                  | 协议 | 职责                                                           |
| :-------------------- | :--- | :------------------------------------------------------------- |
| tRPC (query/mutation) | HTTP | 会话元数据读写（列表、历史、重命名、删除、Fork）、模型列表查询 |

## 4. tRPC API 设计

### 4.1 Procedure 列表

| Procedure          | 类型     | Input                                | 描述                                                           |
| :----------------- | :------- | :----------------------------------- | :------------------------------------------------------------- |
| `sessions.list`    | query    | —                                    | 返回完整会话树结构                                             |
| `sessions.history` | query    | `{ id: string, cursor?: string }`    | 返回指定会话历史消息，支持游标分页                             |
| `sessions.rename`  | mutation | `{ id: string, name: string }`       | 重命名会话                                                     |
| `sessions.delete`  | mutation | `{ id: string }`                     | 删除会话节点（可级联删除子节点）                               |
| `sessions.fork`    | mutation | `{ id: string, messageId?: string }` | Fork 会话，可选截断点                                          |
| `models.list`      | query    | —                                    | 返回内置模型与自定义模型列表（读取 `~/.pi/agent/models.json`） |

> tRPC 挂载路径：`/trpc`，通过 `createExpressMiddleware` 接入 Express v5。

### 4.2 Router 骨架

```ts
// src/shared/trpc.ts
import { initTRPC } from "@trpc/server";
import superjson from "superjson";

export const t = initTRPC.create({ transformer: superjson });
export const router = t.router;
export const publicProcedure = t.procedure;
```

```ts
// src/domain/sessions/router.ts
import { z } from "zod";
import { router, publicProcedure } from "../../shared/trpc.js";

export const sessionsRouter = router({
  list: publicProcedure.query(async () => {
    /* 读 session-map.json */
  }),
  history: publicProcedure
    .input(z.object({ id: z.string(), cursor: z.string().optional() }))
    .query(async ({ input }) => {
      /* 读 messages.jsonl，游标分页 */
    }),
  rename: publicProcedure
    .input(z.object({ id: z.string(), name: z.string() }))
    .mutation(async ({ input }) => {
      /* 更新 session-map.json */
    }),
  delete: publicProcedure.input(z.object({ id: z.string() })).mutation(async ({ input }) => {
    /* 删除会话目录 */
  }),
  fork: publicProcedure
    .input(z.object({ id: z.string(), messageId: z.string().optional() }))
    .mutation(async ({ input }) => {
      /* Fork 会话 */
    }),
});
```

```ts
// src/app.ts — 挂载
import { createExpressMiddleware } from "@trpc/server/adapters/express";
import { appRouter } from "./router.js";

app.use("/trpc", createExpressMiddleware({ router: appRouter }));
```

**类型导出（给前端共享）：**

```ts
// src/expose.ts
import type { AppRouter } from "./router.js";
export type { AppRouter };
```

前端通过 `import type { AppRouter } from '@divisor-agent/server/router'` 引入。

### 4.3 数据结构

#### SessionNode

```ts
interface SessionNode {
  id: string;
  parentId: string | null;
  name: string;
  timestamp: number;
  children?: SessionNode[];
}
```

#### HistoryMessage

```ts
interface HistoryMessage {
  id: string;
  sessionId: string;
  role: "user" | "assistant";
  blocks: MessageBlock[];
  timestamp: number;
}

type MessageBlock =
  | { type: "text"; content: string }
  | { type: "thinking"; content: string }
  | { type: "tool_result"; toolName: string; content: string };
```

> 注：针对于用户发送带有 `@某个文件` 的富文本（TipTap 提供），前端需要将其序列化为原生的 JSX/XML 占位标签（如 `<file path="...">`）并以普通的 `text` 块形式提交与持久化，以此保留其在原句中的语境位置不变。

#### 会话历史响应

```ts
interface HistoryResponse {
  messages: HistoryMessage[];
  nextCursor: string | null;
}
```

### 4.4 错误处理

tRPC 使用 `TRPCError` 替代手写 ApiError：

```ts
import { TRPCError } from "@trpc/server";

throw new TRPCError({
  code: "NOT_FOUND", // 对应 HTTP 404
  message: "Session not found",
});
```

常用错误码映射：`NOT_FOUND` / `BAD_REQUEST` / `INTERNAL_SERVER_ERROR` / `UNAUTHORIZED`。前端自动获得类型化的错误对象，无需额外定义 `ApiError` 接口。

## 5. 会话存储设计

### 5.1 目录结构

```
~/.pi-agent/
├── session-map.json           # 会话树元数据
└── sessions/
    ├── {sessionId}/
    │   ├── meta.json          # 单会话元数据（name, parentId, timestamp）
    │   └── messages.jsonl     # 消息历史（追加写入）
```

### 5.2 session-map.json 格式

```json
{
  "version": "1",
  "sessions": [
    {
      "id": "sess_001",
      "parentId": null,
      "name": "Initial Session",
      "timestamp": 1700000000000
    }
  ]
}
```

### 5.3 Fork 实现逻辑

1. 读取父会话 `messages.jsonl`，按 `messageId` 截断（POI Fork）或完整复制（全局 Fork）。
2. 为新会话分配 `newSessionId`，写入 `session-map.json`（`parentId` 指向父会话）。
3. 将截断后的消息写入新会话 `messages.jsonl`。
4. 新会话的后续消息与父会话**物理隔离**，父会话不受影响。

### 5.4 一致性要求

- `session-map.json` 写入使用原子替换（先写临时文件，再重命名）。
- 消息追加使用 `messages.jsonl`（每行一条 JSON），避免全量读写。
- 版本号字段用于未来协议演进兼容。

## 6. 安全设计

### 6.1 能力边界

Server 只负责：

- 会话元数据的增删改查
- 历史消息的读写
- 模型配置读取

Server 不负责：

- 工具执行（App 端负责）
- 路径沙箱（App 端负责）
- 命令过滤（App 端负责）

### 6.2 审计日志

- 每次会话创建、删除、Fork 均写入结构化日志。
- 日志字段：`sessionId`、`operation`、`timestamp`、`result`。
- 日志按会话 ID 可检索。
